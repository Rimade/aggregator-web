## План проекта: агрегатор заказов по QR (MVP, оплата на месте)

Этот документ — **единая “истина”** о том, что мы строим, в каком порядке, и какие правила безопасности/изоляции соблюдаем.

### Цель MVP

- **Что делаем**: QR-меню + оформление заказа + панель заказов для сотрудников кафе.
- **Чего НЕ делаем в MVP**: доставка, онлайн-оплата, интеграции iiko/r_keeper, фискализация, мобильные приложения.
- **Монетизация**: бизнес платит за доступ, **вы вручную добавляете** бизнес в систему и выдаёте доступ сотрудникам.

---

## 1) Роли и доступ

### Роли

- **platform_admin (вы)**: создаёт кафе, управляет пользователями кафе, может смотреть всё.
- **cafe_staff (сотрудники кафе)**: видят и изменяют только данные своего кафе.
- **customer (клиент)**: без логина, может смотреть меню и создавать заказ для конкретного кафе.

### Принцип изоляции (multi-tenant)

Все данные “принадлежат” конкретному кафе через `cafe_id`.

**Жёсткое правило**:

- Любой запрос из админки должен быть ограничен `cafe_id` текущего пользователя.
- Клиентский доступ (витрина) ограничен `cafe_id` из URL (slug).

**Техническая реализация изоляции (выбираем Supabase):**

- Таблица `profiles` связывает `auth.users.id -> cafe_id + role`.
- Включаем RLS на “чувствительных” таблицах.
- Политики:
  - `platform_admin`: полный доступ.
  - `cafe_staff`: доступ только к строкам с `cafe_id = profiles.cafe_id`.

## 2) Технологии (выбрано под скорость)

### Frontend + “тонкий сервер”

- **Next.js + TypeScript**
- **TailwindCSS**
- **React Hook Form + Zod**
- **TanStack Query** (по желанию)

### Backend-as-a-service

- **Supabase**:
  - Postgres (данные)
  - Auth (логин сотрудников)
  - Storage (картинки блюд)
  - RLS (изоляция кафе)

### Почему без отдельного “полного бэка”

В MVP (оплата на месте) нет критичных серверных задач (webhook платежей, секреты эквайринга).
Мы используем Supabase как бэкенд, а Next.js — как UI + минимум серверных маршрутов при необходимости.

## 3) MVP-функции (скоп)

### Публичная витрина по QR

- Страница кафе (по `slug`)
- Каталог: категории, блюда, модификаторы (опционально), цены
- Корзина
- Оформление заказа:
  - режим: **в зале** (со столом) или **самовывоз**
  - имя/телефон (опционально) + комментарий
- Экран успеха: номер заказа + состав + сумма + статус “принят”
- Оплата: **только на месте**

### Панель кафе (админка кафе)

- Вход/выход
- Заказы: список + смена статуса:
  - `new` → `in_progress` → `ready` → `closed`
  - `cancelled` (опционально)
- Управление меню:
  - категории
  - позиции
  - цены
  - фото (Storage)
- Настройки кафе:
  - название, адрес (опц.), часы работы
  - включить/выключить приём заказов

### Панель платформы (вы)

- Создать кафе
- Создать сотрудника кафе / сбросить пароль / отключить
- Управление подпиской/оплатой доступа (в MVP можно просто флаг `is_active`)

## 4) Модель данных (черновик схемы)

### Supabase Auth

- `auth.users` — сотрудники (логин по email).

### Таблицы приложения

- `cafes`
  - `id` (uuid)
  - `name`
  - `slug` (уникальный)
  - `is_active` (bool) — доступ оплачен/включён
  - `accepting_orders` (bool)
  - `created_at`

- `profiles`
  - `user_id` (uuid, = auth.users.id)
  - `role` (`platform_admin` | `cafe_staff`)
  - `cafe_id` (uuid, nullable для platform_admin)

- `menu_categories`
  - `id`, `cafe_id`, `name`, `sort_order`

- `menu_items`
  - `id`, `cafe_id`, `category_id`
  - `name`, `description`, `price`
  - `is_available`
  - `image_path` (опционально)

- `orders`
  - `id`, `cafe_id`
  - `order_number` (читаемый номер, уникальный в рамках cafe)
  - `type` (`dine_in` | `pickup`)
  - `table_label` (опц.)
  - `customer_name` (опц.)
  - `customer_phone` (опц.)
  - `comment` (опц.)
  - `status` (`new` | `in_progress` | `ready` | `closed` | `cancelled`)
  - `total_amount`
  - `created_at`

- `order_items`
  - `id`, `order_id`, `cafe_id`
  - `menu_item_id`
  - `name_snapshot` (фиксируем название на момент заказа)
  - `price_snapshot`
  - `qty`

Примечание: `*_snapshot` поля важны, чтобы изменения меню не “ломали” старые заказы.

## 5) Правила доступа (RLS-политики, требования)

### Чтение витрины (клиент без логина)

- Разрешаем читать `cafes` (только активные) по `slug`.
- Разрешаем читать категории/позиции по `cafe_id`, но только если кафе активно и принимает заказы.

### Создание заказа (клиент без логина)

- Разрешаем `insert` в `orders` и `order_items` только если:
  - кафе активно (`cafes.is_active = true`)
  - приём заказов включён (`accepting_orders = true`)
  - `cafe_id` задан
- Запрещаем клиенту изменять статусы заказов.

### Админка кафе (cafe_staff)

- Полный CRUD по меню своего `cafe_id`.
- Чтение/обновление заказов своего `cafe_id`.

### Платформенный админ (platform_admin)

- Может всё (или через service-role ключ на серверной стороне).

## 6) Потоки (flows)

### Добавление нового кафе (вы)

1. Создать `cafes` запись (`name`, `slug`, `is_active=true`).
2. Создать сотрудника в Supabase Auth (email + invite).
3. Создать `profiles` для сотрудника с `role=cafe_staff` и `cafe_id`.
4. Сгенерировать QR на URL витрины: `/c/{slug}`.
5. (Опционально) QR для столов: `/c/{slug}?t=7` (или `/c/{slug}/t/7`).

### Заказ клиентом (QR)

1. Клиент открывает витрину кафе по `slug`.
2. Собирает корзину, отправляет заказ.
3. В БД создаётся `orders` + `order_items`.
4. Кафе видит новый заказ в панели.

## 7) Порядок реализации (чёткие шаги)

### Шаг A — инициализация проекта

- Создать Next.js (TS) + Tailwind.
- Подключить Supabase SDK.

### Шаг B — БД + RLS

- Создать таблицы.
- Настроить RLS и политики (критично для изоляции).
- Создать `platform_admin` профиль.

### Шаг C — публичная витрина

- Роут `/c/[slug]`:
  - загрузка кафе + меню
  - корзина + форма заказа
  - создание заказа в БД

### Шаг D — админка кафе

- Логин
- Страницы:
  - заказы (канбан/таблица)
  - меню (категории/позиции)
  - настройки кафе

### Шаг E — панель платформы

- “Создать кафе”
- “Создать сотрудника”
- “Показать QR”

## 8) Решения “чтобы было безопасно для вас”

- Деньги **не проходят через платформу** (в MVP).
- Платформа — сервис формирования/передачи заказов.
- “Доступ оплачен” = флаг `cafes.is_active`.
- В документах позже: договор/оферта и политика ПДн (после MVP).

---

## 9) Критерии готовности MVP

- Можно добавить кафе и сотрудника.
- Сотрудник видит только своё кафе и его заказы/меню.
- Клиент по QR видит меню конкретного кафе и создаёт заказ.
- Заказ появляется у сотрудников и меняет статус.
